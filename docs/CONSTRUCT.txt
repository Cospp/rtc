# Streaming Platform – Architecture README (MVP → Production-Nah)

## 1. Ziel

Diese Plattform soll mehrere eingehende Streams von Clients annehmen, diese deterministisch an dedizierte Worker koppeln und den Plattformzustand extern in Redis verwalten.

Der Fokus liegt auf einer **production-nahen Grundplattform**, nicht auf einem schnellen Hack.

### Primäre Architekturziele

* Trennung von **Control Plane** und **Media Plane**
* **1 Worker = 1 aktive Session** zur Reduktion des Blast Radius
* externer Zustandsstore mit **TTL / Leases / Heartbeats**
* spätere Erweiterbarkeit auf **ICE / STUN / TURN**
* lokale Ausführung, aber mit **Cloud-übertragbarem** Schnitt
* gute Observability mit **Prometheus-Metriken**

### Nicht-Ziele für den ersten MVP

* vollständiges WebRTC von Anfang an
* TURN in der ersten Iteration
* echter Stream-Handover zwischen Workern
* persistente Business-Datenbank
* komplexe Authentifizierung / Mandantenfähigkeit

## 3. Kernprinzipien

### 3.1 Control Plane vs Media Plane

**Control Plane**

* Session annehmen
* Worker auswählen
* atomar reservieren
* Session-Zustand verwalten
* Assignment an Client zurückgeben

**Media Plane**

* Stream annehmen
* Stream verarbeiten
* Media-spezifische Fehler behandeln
* Streaming-Lifecycle melden

Diese Trennung ist nicht optional. Sie ist Kern des Designs.

### 3.2 Isolation

Ein Worker nimmt genau **eine** aktive Session an.

**Begründung:**

* kleinere Failure Domain
* klarere Verantwortlichkeit
* einfachere Analyse bei Ausfällen
* safety-orientiertes Modell

**Trade-off:**

* schlechtere Ressourcenausnutzung
* höherer Pod-/Worker-Overhead
* mehr Scheduling- und Allocation-Aufwand

### 3.3 Redis ist Koordinationsstore, nicht Medienpfad

Redis hält:

* Worker-Zustände
* Session-Zustände
* TTL / Leases
* Heartbeats
* Mapping `client -> session -> worker`

Redis hält **nicht**:

* Medienpayload
* Videoframes
* RTP/UDP-Daten
* langfristige Geschäftsobjekte

## 4. Komponenten

### 4.1 Session Control

#### Verantwortung

* `POST /sessions` entgegennehmen
* Request validieren
* freien Worker finden
* atomar reservieren
* Session in Redis anlegen
* `connection_info` + Token zurückgeben

#### Muss stateless sein

Die Komponente darf keine kritische Session-Wahrheit nur im RAM halten.

#### Darf nicht

* Streamdaten terminieren
* Worker-Lebenszyklus global alleine „wissen"
* Redis als alleinige Wahrheit ohne TTL-/Lease-Modell missbrauchen

---

### 4.2 Worker

#### Verantwortung

* Registrierung in Redis beim Start
* Heartbeats senden
* Session-Token validieren
* Stream annehmen
* Session als `streaming` markieren
* bei Ende / Fehler Zustand zurückschreiben

#### Regeln

* ein Worker akzeptiert genau eine aktive Session
* Worker geht nach Session-Ende zurück auf `warm` oder wird bewusst ersetzt

---

### 4.3 Redis

#### Verantwortung

* flüchtige Registry
* Session-/Worker-Mapping
* TTL / Lease
* Cleanup über Ablaufzeiten

#### Risiken

* stale Einträge
* divergierende Indizes
* doppelte Reservation ohne atomare Operationen

Daher muss es atomare Reservation geben.

---

### 4.4 Client Simulator

#### Verantwortung

* N Clients simulieren
* Session anfragen
* Assignment konsumieren
* UDP-Stream senden
* Lastprofile variieren
* Failure-Cases simulieren

#### Parameter

* Anzahl paralleler Clients
* Paketgröße
* Frame-/Sendeintervall
* Dauer
* Streamprofil (z. B. 480p)

## 6. Zustandsmodelle

### 6.1 Worker State Machine

```text
starting -> warm -> reserved -> active -> warm
                  \-> dead
active -> dead
warm   -> dead
```

#### Zustände

* `starting`: noch nicht zuweisbar
* `warm`: frei und sofort zuweisbar
* `reserved`: Session zugewiesen, wartet auf Bind
* `active`: Stream läuft
* `dead`: Heartbeat weg / Crash / ungültig
* optional später: `draining`

### 6.2 Session State Machine

```text
requested -> assigned -> connecting -> streaming -> closed
                           \-> failed
streaming -----------------> failed
```

#### Zustände

* `requested`
* `assigned`
* `connecting`
* `streaming`
* `failed`
* `closed`

Später erweiterbar um `reconnecting`.

## 8. Datenfluss (Happy Path)

### 1. Worker startet

* registriert sich in Redis
* Status `starting`
* initialisiert Endpoint
* Heartbeat startet
* Status wird `warm`
* Worker wird in `workers:warm` aufgenommen

### 2. Client fordert Session an

Client ruft `POST /sessions` auf.

### 3. Session Control reserviert Worker atomar

* Worker aus `workers:warm` auswählen
* prüfen, ob Worker noch `warm` ist
* Worker auf `reserved` setzen
* Session anlegen
* Mapping `client -> session -> worker` schreiben

### 4. Assignment zurückgeben

Client erhält:

* `session_id`
* `worker_id`
* `token`
* `ttl_seconds`
* `connection_info`

### 5. Client verbindet sich zum Worker

Client sendet Handshake mit `session_id` und `token`.

### 6. Worker validiert und aktiviert Session

* Session gültig?
* Worker-ID passt?
* Token gültig?
* Session noch nicht abgelaufen?

Dann:

* Session `connecting -> streaming`
* Worker `reserved -> active`

### 7. Stream läuft

* Worker sendet Heartbeats
* Session TTL wird gepflegt
* Metriken werden aktualisiert

### 8. Stream endet

* Session `closed`
* Worker zurück auf `warm`
* Cleanup der Indizes

## 9. Timeouts / TTL / Leases

### Worker TTL

* kurzer TTL-Wert
* wird über Heartbeat verlängert
* bei Ablauf gilt Worker als `dead`

### Reservation TTL

* Session in `assigned/connecting` ist kurzlebig
* falls Client nicht bindet, wird Reservation freigegeben

### Streaming TTL

* während aktivem Stream verlängerbar
* bei Heartbeat-/Keepalive-Verlust läuft Session aus

### Empfehlung für die erste Iteration

* Worker Heartbeat: alle 5s
* Worker TTL: 15s
* Reservation TTL: 20–30s
* Session Keepalive/Progress: alle 5–10s

Exakte Werte sind noch Feintuning, aber die Mechanik muss von Anfang an existieren.

---

## 10. Atomare Reservation

### Problem

Zwei Clients dürfen niemals denselben Worker zugewiesen bekommen.

### Lösung

Reservation muss atomar sein, z. B. über:

* Redis Lua Script
* Compare-and-set-ähnliche Logik
* konsistente Updates von:

  * `workers:warm`
  * `worker:{id}`
  * `session:{id}`

Ohne atomare Reservation ist das System fachlich inkorrekt.

## 12. Erweiterbarkeit Richtung ICE / STUN / TURN

Diese Plattform darf später nicht daran scheitern, dass heute `connection_info` zu simpel modelliert wurde.

### Heute

```json
{
  "transport": "udp",
  "host": "worker-07.local",
  "port": 50000
}
```

### Später denkbar

```json
{
  "transport": "webrtc",
  "ice": {
    "ufrag": "...",
    "pwd": "...",
    "candidates": ["..."],
    "stun_servers": ["stun:stun.example.com:3478"],
    "turn_servers": ["turn:turn.example.com:3478"]
  }
}
```

Daher ist `connection_info` ein strukturierter Container und kein primitiver String.

## 14. Failure Modes

### Kein Worker verfügbar

* `POST /sessions` liefert kontrollierten Fehler
* Metrik wird erhöht

### Worker stirbt nach Reservation

* Session bleibt in `connecting`
* TTL läuft aus
* Worker wird `dead`
* Client muss neu anfragen

### Client bindet nie

* Reservation läuft automatisch aus
* Worker wird wieder freigegeben

### Heartbeat fehlt

* Worker wird als `dead` betrachtet
* Session wird `failed`, falls aktiv

### Redis-State divergiert

* später braucht es Reconciliation
* im MVP reicht dokumentierte Schwäche + einfache Cleanup-Strategie

---

## 15. Bekannte Schwächen des MVP

* Redis ist zentraler Koordinationspunkt
* 1 Worker pro Session ist ressourcenintensiv
* direkter Worker-Endpoint ist noch nicht die endgültige Cloud-Lösung
* Reconnect ist zunächst Neuallokation, kein echtes Handover
* vereinfachtes UDP ist noch kein vollwertiges RTC

Diese Schwächen sind bewusst akzeptiert und müssen im Architekturgespräch benannt werden.

## 17. Offene Architekturentscheidungen

Diese Punkte müssen vor dem Coden explizit entschieden werden:

1. **Worker-Exposure-Modell**

   * direkte Worker-Adresse
   * dedizierter Service pro Worker
   * vorgeschalteter Media-Gateway

2. **Worker-Lifecycle**

   * Worker nach Session-Ende wieder `warm`
   * oder Worker beenden und neu starten

3. **Handshake-Protokoll**

   * einfacher UDP-Handshake
   * oder separater TCP/HTTP-Setup am Worker

4. **Reconnect-Semantik**

   * reine Neuallokation
   * oder vorbereitete Session-Recovery

5. **Cleanup / Reconciliation**

   * rein TTL-basiert
   * oder zusätzlicher Reconciliation-Loop

## 19. Gesprächsrelevante Architekturargumente

Dieses Projekt sollte im Gespräch zeigen können:

* warum **Control Plane** und **Media Plane** getrennt sind
* warum **1 Worker = 1 Stream** den Blast Radius reduziert
* warum Redis nur **Koordinationsstore** ist
* warum TTL/Heartbeats für verteilte Zustände essenziell sind
* warum atomare Reservation wichtiger ist als reine Funktionalität
* warum das Design später auf ICE/STUN erweitert werden kann
* welche Schulden bewusst akzeptiert wurden
